How to Build Things

April 14, 2026

Building something—a product, a feature, a side project—follows a pattern. It's not complicated if you ignore all the noise.

Start with a Real Problem

The best products solve problems people actually have. Spend time understanding the problem before you code. Talk to people. Write it down. Make sure this is something worth solving.

"The most dangerous moment is when you have a solution looking for a problem."

Make It Simple First

Don't start with architecture. Don't start with perfect abstractions. Start with the simplest thing that could possibly work. Build it, even if it's ugly. You'll learn more from running code than planning.

Perfect is the enemy of shipped. Ship something real, get feedback, iterate from there.

Ship Early

Get it in front of people. Real usage beats all the internal testing you can do. People will use it in ways you didn't expect. They'll break it. They'll ask for things you didn't think of. That's the feedback loop.

The act of shipping teaches you more than months of planning.

Stay Focused

Every feature is debt. Every polish takes time. Cut ruthlessly. What's the core thing that makes this valuable? Do that. Cut everything else. You can always add more later, but shipping with less is better than shipping late with more.

Iterate

First version won't be perfect. That's fine. Get feedback, fix the big things, ship again. Each iteration makes it better. The compounding effect of small iterations adds up fast.

This is how you build. Not perfectly from day one, but iteratively, with real feedback, shipping constantly.
